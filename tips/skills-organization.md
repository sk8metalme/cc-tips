# Skills の整理方法

Claude Code の Skills に何を置くべきかの判断基準。

## 基本原則：「考え方」と「成果物」で分類

| 分類 | 置き場所 | 性質 | 例 |
|------|----------|------|-----|
| 考え方・ガイドライン | Skills | 再利用可能 | 「画面をこう設計すべき」というルール |
| 成果物・ドキュメント | docs/ や specs/ | コンテキスト固有 | 「この画面はこう設計した」という結果 |

## 判断基準

**再利用可能かどうか**が最も重要な判断基準。

### Skills に向いているもの

- 画面設計時のチェックリスト
- UIコンポーネントの命名規則
- 状態管理の設計パターン
- コードレビューの観点リスト
- エラーハンドリングのガイドライン

### Skills に向いていないもの

- プロジェクト固有の設計判断（「このプロジェクトではAPIをRESTにした」という決定記録）
- 生の調査・実験ログ（「○○を試した結果、△△だった」という一時的な記録）

**ボーダーラインケースの判断:**

迷ったときは以下を基準に判断してください：
- **中央集約的で、再利用可能で、恒久的な知識** → Skills に置く
- **一時的な内容や特定プロジェクトに密接に結合している内容** → Skills に置かず、プロジェクトの docs/ に置いてリンク参照

例：「ログイン画面の設計」は、一般的な設計原則（Skills向き）と、このプロジェクトの具体的な実装（docs/向き）を分けて管理する。

## なぜこの区別が重要か

Skills に成果物を置くと：
- 特定プロジェクトの文脈が他で邪魔になる
- 更新のたびに Skills を編集する必要がある

docs/ に考え方を置くと：
- 複数プロジェクトで同じ内容を重複管理
- ルールの一貫性が保てなくなる
